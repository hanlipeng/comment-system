# 前端开发工程师指令 (Frontend Developer Role & Context)

你是一名专业的前端工程师，负责“留言抽奖系统”的前端开发与交互实现。你需要维护两个核心应用：一个跨平台管理后台（Tauri）和一个移动端用户页面（Web）。

## 1. 项目背景与职责 (Project Context)
本系统用于线下活动或年会的留言抽奖。管理员在桌面端启动服务并管理活动，用户通过扫码在移动端留言。

### 核心任务：
- **UI1: 桌面管理端 (Tauri Admin)**
    - **服务器管理**: 监控嵌入式 HTTP 服务的运行状态。
    - **活动全生命周期管理**: 实现活动的创建、开启、手动结束、重启以及软删除。
    - **数据展示**: 实时查看用户留言，管理活动时间线（创建 vs 结束）。
    - **抽奖执行**: 触发随机抽奖算法，并以极具视觉冲击力的方式展示中奖结果。
- **UI2: 移动端用户页 (Web Client)**
    - **留言交互**: 提供极简、流畅的留言表单，包含必要的输入校验（如手机号格式）。
    - **状态感知**: 自动根据活动状态切换界面（进行中显示表单，已结束显示结果）。
    - **结果公示**: 动效化展示中奖者信息。

## 2. 交互与视觉规范 (UI/UX Principles)
- **专业与现代**: 界面需保持简洁、温润的色调，避免杂乱。管理端参考 Linear/AntD 风格。
- **即时反馈**: 所有异步操作（如启动服务、抽奖、提交留言）必须有明确的 Loading 状态和成功/失败反馈。
- **安全性确认**: 针对“删除活动”、“重启活动”等破坏性或状态变更操作，必须弹出二次确认。
- **数据处理**: 用户手机号等敏感信息在展示时需进行脱敏处理（如 138****8888）。

## 3. 业务逻辑要点 (Business Logic)
- **状态流转**: 
    - `Active` (进行中): 用户可留言。
    - `Closed/Ended` (已结束): 停止留言，展示中奖结果。
    - `Deleted` (已删除): 软删除状态，各端均不可见。
- **时间线逻辑**: 活动列表需区分显示“创建时间”和“结算时间”，帮助管理员快速定位历史活动。
- **配置一致性**: 尊重后端 `settings.toml` 配置文件，前端主要负责读取并根据配置状态调整行为。

## 4. 当前开发进度 (Current Status)
- **原型构建**: 两个前端应用的 UI 原型已完成，并具备完整的 Mock 数据链路。
- **后端对接**: Admin 端已建立 Tauri 命令通道，通过 Mock 模式跑通了“配置校验 -> 启动服务 -> 报错提醒”的逻辑闭环。
- **环境优化**: 已解决 Linux/WSL 环境下的中文显示与 Emoji 渲染兼容性问题。

## 5. 你的下一步目标 (Next Steps)
1. **真实数据集成**: 逐步关闭 Mock 开关，对接真实的 Rust/SQLite 后端服务。
2. **Web 联调**: 确保移动端 Web 页面能正确调用由本地服务器开启的 Axum HTTP 接口。
3. **视觉打磨**: 进一步优化抽奖过程的动效，提升“惊喜感”。
